## The ModelRoute config system

ModelRoute uses a **YAML-first** configuration system. Providers, models, pricing, and request/response mappings are all defined in YAML files. This means:

- **No custom code per provider** вЂ” every provider is config-driven
- **Instant validation** вЂ” `make validate-configs` checks everything without a database
- **Hot-reload** вЂ” changes take effect without redeployment
- **Version control** вЂ” every config change is a git commit with full audit trail

## Quick start: add a new provider in 5 minutes

```bash
# 1. Scaffold the provider skeleton
make scaffold-provider NAME=replicate BASE_URL=https://api.replicate.com AUTH=bearer SECRET=replicate_api_key

# 2. Scaffold a model
make scaffold-model PROVIDER=replicate MODEL=sdxl CATEGORY=image-generation MODE=ASYNC

# 3. Edit the generated configs (fill in real endpoints, body templates, pricing)
$EDITOR configs/providers/replicate/config.yaml

# 4. Validate everything
make validate-configs

# 5. Start the dev server (hot-reload enabled)
make dev

# 6. Reload configs (if already running)
curl -X POST http://localhost:8080/api/v1/admin/providers/reload
```

## Workflow

<Steps>
  <Step title="Scaffold">
    Generate the YAML skeleton with `make scaffold-provider` and `make scaffold-model`. This creates the directory structure and fills in sensible defaults.
  </Step>
  <Step title="Configure">
    Edit the YAML to add real API endpoints, authentication, body templates, input schemas, pricing, and response mappings. See the detailed guides below.
  </Step>
  <Step title="Validate">
    Run `make validate-configs` to check all provider configs, model definitions, and translation mappings. This catches errors without needing a database or network.
  </Step>
  <Step title="Test locally">
    Run `make dev` to start the CRM service locally. Configs hot-reload on file save (fsnotify). Make real API calls against `http://localhost:8080`.
  </Step>
  <Step title="Deploy">
    Commit your configs to git and push. The deploy pipeline picks up the changes automatically.
  </Step>
</Steps>

## Admin API

For runtime operations that don't require config file changes:

| Endpoint | Method | Description |
|-|-|-|
| `/api/v1/admin/providers` | GET | List all loaded providers |
| `/api/v1/admin/providers/reload` | POST | Trigger config hot-reload |
| `/api/v1/admin/providers/status` | GET | Reload status + last reload time |
| `/api/v1/admin/routes` | GET/POST | List/create routing rules |
| `/api/v1/admin/routes/:id` | GET/PUT/DELETE | Manage a specific route |
| `/api/v1/admin/routes/:id/providers` | GET/POST | Manage route providers |
| `/api/v1/admin/routes/:id/pricing` | GET/POST | Runtime pricing overrides |
